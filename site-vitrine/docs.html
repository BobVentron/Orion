Présentation de la stack

Orion est distribuée sous forme d'un paquet Debian qui pilote une stack Docker composée de quatre conteneurs :

Conteneur Rôle Port
orion_web Interface web (Nginx + frontend statique) 80
orion_api API REST Node.js 3000 (interne)
orion_clt Module de collecte réseau (CLT) —
orion_db Base de données PostgreSQL 16 5432 (interne)
ℹ
Architecture réseau Les conteneurs orion_db et orion_api ne sont pas exposés sur l'hôte. Seul orion_web est accessible depuis l'extérieur.

Installation

Procédure complète d'installation sur un serveur Debian/Ubuntu vierge. Durée estimée : 10 à 15 minutes.

⚠
Accès root requis L'installation nécessite des droits sudo. Assurez-vous d'être connecté avec un compte disposant des privilèges d'administration.
1
Vérifier les prérequis système
Avant tout, confirmez que le serveur remplit les conditions minimales (distribution, RAM, espace disque).
bash 
lsb_release -a
free -h
df -h /
2
Installer Docker Engine
Si Docker n'est pas présent sur le serveur, installez-le via le script officiel. Si Docker est déjà installé, passez à l'étape 3.
bash 
# Installation de Docker Engine (script officiel)
curl -fsSL https://get.docker.com | sudo sh

# Optionnel : ajouter l'utilisateur courant au groupe docker
# (évite de taper sudo à chaque commande docker)
sudo usermod -aG docker $USER
newgrp docker

# Vérifier l'installation
docker --version && docker compose version
3
Ajouter le dépôt Orion
Enregistrez le dépôt APT d'Orion, puis installez le paquet.
bash 
# Ajouter le dépôt
echo "deb [trusted=yes] https://orion.duckinc.soleilhac.me/packages/ ./" \
  | sudo tee /etc/apt/sources.list.d/orion.list

# Installer le paquet
sudo apt update && sudo apt install orion
4
Vérifier la configuration et déployer
orion-control check valide l'environnement avant de lancer la stack. Ensuite, orion-control deploy génère le mot de passe PostgreSQL et démarre les quatre conteneurs.
bash 
# Valider l'environnement
orion-control check

# Déployer la stack
orion-control deploy
5
Vérifier le bon fonctionnement
Attendez quelques secondes que PostgreSQL termine son initialisation, puis vérifiez l'état des conteneurs et l'API.
bash 
# État des conteneurs (orion_db doit être 'healthy')
orion-control status

# Tester le health check de l'API
curl http://localhost:3000/health
# Résultat attendu : {"status":"healthy","database":"connected",...}

# Accéder à l'interface web
# Ouvrir http://<IP_DU_SERVEUR>/ dans un navigateur
✓
Installation réussie Si les quatre conteneurs sont Up et que l'API retourne {"status":"healthy"}, Orion est opérationnel.

orion-control

L'outil en ligne de commande pour piloter l'ensemble du cycle de vie de la stack Orion.

Référence des commandes

CommandeDescription
orion-control status Affiche l'état de chaque conteneur (Up, healthy, exited…)
orion-control deploy Démarre la stack (génère le mot de passe si premier lancement)
orion-control check Valide l'environnement avant déploiement
orion-control update Met à jour les images Docker et redémarre les conteneurs
orion-control logs [service] Affiche les logs d'un ou tous les conteneurs
orion-control clear Arrête les conteneurs et supprime les volumes Destructif
✕
orion-control clear Cette commande supprime tous les volumes Docker, incluant la base de données PostgreSQL. Elle entraîne une perte de données irréversible si aucune sauvegarde n'a été effectuée.

Exemples d'utilisation

bash 
# Voir l'état de tous les conteneurs
orion-control status

# Suivre les logs de l'API en temps réel
orion-control logs api

# Suivre les logs de tous les services
orion-control logs

# Relancer la stack après une mise à jour du paquet
orion-control update

Configuration

Les paramètres de la stack Orion sont centralisés dans un seul fichier d'environnement.

Fichier de configuration

Le fichier de configuration principal est /etc/orion/.env. Il est généré automatiquement lors du premier orion-control deploy.

env 
# /etc/orion/.env

# Identifiants PostgreSQL (générés automatiquement)
POSTGRES_USER=orion_admin
POSTGRES_PASSWORD=xxxxxxxxxxxxxxxx
POSTGRES_DB=orion_db

# Port d'écoute de l'interface web
ORION_WEB_PORT=80

# Niveau de log de l'API (debug | info | warn | error)
LOG_LEVEL=info
⚠
Modifications du mot de passe Si vous modifiez POSTGRES_PASSWORD après le premier déploiement, vous devez aussi mettre à jour le mot de passe directement dans PostgreSQL via docker exec, sinon l'API ne pourra plus se connecter.

Appliquer les modifications

Tout changement dans /etc/orion/.env nécessite un redémarrage de la stack pour être pris en compte :

bash 
sudo nano /etc/orion/.env

# Redémarrer après modification
orion-control deploy

Maintenance & mises à jour

Procédures à suivre pour la vérification de santé quotidienne et les mises à jour d'Orion.

Vérification de santé quotidienne

Ces vérifications peuvent être intégrées dans un cron ou exécutées lors des rondes d'administration :

bash 
#!/bin/bash
# Script de vérification santé Orion

echo "=== État des conteneurs ==="
orion-control status

echo ""
echo "=== Health check API ==="
HTTP_CODE=$(curl -s -o /dev/null -w "%{http_code}" http://localhost:3000/health)
if [ "$HTTP_CODE" = "200" ]; then
    echo "✓ API opérationnelle (HTTP 200)"
else
    echo "✗ API indisponible (HTTP $HTTP_CODE)"
fi

echo ""
echo "=== Taille de la base de données ==="
docker exec orion_db psql -U orion_admin -d orion_db -c \
    "SELECT pg_size_pretty(pg_database_size('orion_db')) AS taille;"

echo ""
echo "=== Logs d'erreur récents (API) ==="
docker logs --tail=20 orion_api 2>&1 | grep -i error || echo "Aucune erreur récente"

Procédure de mise à jour

Les mises à jour suivent un processus en deux temps : mise à jour du paquet Debian, puis mise à jour des images Docker. La base de données n'est pas affectée car les données persistent dans un volume Docker.

1
Mettre à jour le paquet Debian
bash 
sudo apt update && sudo apt upgrade orion

# Vérifier la nouvelle version installée
dpkg -l orion
2
Mettre à jour les images Docker et redémarrer
bash 
orion-control update
3
Vérifier le bon fonctionnement
bash 
orion-control status
curl http://localhost:3000/health
ℹ
Migrations de schéma Si une mise à jour majeure inclut des modifications de schéma de base de données, des étapes supplémentaires seront documentées dans les notes de version associées.

Sauvegarde & restauration

Orion ne dispose pas de mécanisme de sauvegarde automatique intégré. La sauvegarde PostgreSQL est de la responsabilité de l'administrateur système.

⚠
Obligation opérationnelle La sauvegarde régulière est indispensable. Elle doit être intégrée dans votre infrastructure existante (cron + rsync, Veeam, Bacula, etc.).

Sauvegarde avec pg_dump

La méthode recommandée utilise pg_dump exécuté via docker exec. Elle produit un dump cohérent grâce aux verrous PostgreSQL posés pendant l'opération.

bash 
# Sauvegarde format SQL texte
docker exec orion_db pg_dump -U orion_admin orion_db > \
    /backup/orion_$(date +%Y%m%d_%H%M%S).sql

# Sauvegarde compressée (recommandé pour les grandes bases)
docker exec orion_db pg_dump -U orion_admin --format=custom orion_db > \
    /backup/orion_$(date +%Y%m%d_%H%M%S).pgdump

Planification automatique (cron)

bash 
# Sauvegarde quotidienne à 2h00 du matin
# Ajouter dans crontab via : crontab -e
0 2 * * * docker exec orion_db pg_dump -U orion_admin --format=custom orion_db > /backup/orion_$(date +\%Y\%m\%d).pgdump

Restauration

Arrêtez l'API et le module CLT avant la restauration pour éviter des écritures concurrentes.

bash 
# 1. Arrêter les services qui accèdent à la base
docker stop orion_api orion_clt

# 2a. Restauration depuis un dump SQL texte
cat /backup/orion_20260413_020000.sql | \
    docker exec -i orion_db psql -U orion_admin -d orion_db

# 2b. Restauration depuis un dump custom (plus rapide)
docker exec -i orion_db pg_restore -U orion_admin -d orion_db --clean < \
    /backup/orion_20260413_020000.pgdump

# 3. Redémarrer les services
orion-control deploy

Monitoring & logs

Accès aux logs des conteneurs et configuration de la rotation pour éviter une saturation du disque.

Accéder aux logs

Chaque conteneur expose ses logs sur stdout/stderr. orion-control logs est un raccourci vers docker compose logs.

bash 
# Logs de tous les services (50 dernières lignes)
orion-control logs

# Logs d'un service spécifique, en direct
orion-control logs api
orion-control logs web
orion-control logs db

# Accès direct via docker (avec filtres)
docker logs --tail=100 orion_api
docker logs --since 1h orion_api 2>&1 | grep -i error

Configurer la rotation des logs

Sans configuration, Docker stocke les logs sans limite dans des fichiers JSON sur l'hôte. Il est fortement recommandé de limiter leur taille.

json 
// /etc/docker/daemon.json
{
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "10m",
    "max-file": "5"
  }
}
bash 
# Redémarrer Docker pour appliquer la configuration
sudo systemctl restart docker
ℹ
Capacité estimée Avec cette configuration, chaque conteneur est limité à 5 fichiers de 10 Mo, soit 50 Mo par conteneur et environ 200 Mo pour l'ensemble de la stack.

Surveiller la taille de la base

sql 
-- Tables les plus volumineuses
SELECT
    tablename,
    pg_size_pretty(pg_total_relation_size('public.'||tablename)) AS taille_totale
FROM pg_tables
WHERE schemaname = 'public'
ORDER BY pg_total_relation_size('public.'||tablename) DESC
LIMIT 10;

-- Purger les métriques de plus de 90 jours
DELETE FROM device_metrics_history
WHERE time < NOW() - INTERVAL '90 days';

-- Récupérer l'espace disque après purge
VACUUM ANALYZE device_metrics_history;

Diagnostic

Résolution de problèmes

Les problèmes les plus fréquemment rencontrés et leurs solutions.

L'API ne démarre pas — erreur de connexion à PostgreSQL

✕
Symptôme Le conteneur orion_api redémarre en boucle. Les logs affichent des messages du type Tentative X/30 → ECONNREFUSED ou ETIMEDOUT.

Causes et solutions

PostgreSQL n'a pas terminé son initialisation

Attendre quelques secondes et vérifier avec orion-control status. Si orion_db n'est pas encore healthy, c'est normal.

Mot de passe incorrect dans /etc/orion/.env

Vérifier que le mot de passe correspond à celui de la base :

bash 
docker exec orion_db psql -U orion_admin -c '\l'

Volume postgres_data corrompu

Vérifier les logs de orion_db. En dernier recours, réinitialiser (perte de données si pas de sauvegarde) :

bash 
# ⚠ PERTE DE DONNÉES si aucune sauvegarde disponible
orion-control clear
orion-control deploy

Erreur 502 Bad Gateway sur le site web

✕
Symptôme Le site web est accessible mais toutes les requêtes vers /api/ retournent une erreur 502.

Le conteneur orion_web ne peut pas joindre orion_api. Vérifier que l'API est démarrée et healthy :

bash 
orion-control status

# Si l'API est arrêtée, la relancer
docker start orion_api

# Si elle redémarre en boucle, consulter les logs
orion-control logs api

La base de données grossit trop vite

⚠
Symptôme L'espace disque se remplit progressivement, principalement occupé par le volume Docker postgres_data.

Identifier les tables volumineuses puis appliquer une politique de rétention :

sql 
-- Identifier les tables les plus volumineuses
SELECT
    tablename,
    pg_size_pretty(pg_total_relation_size('public.'||tablename)) AS taille_totale
FROM pg_tables
WHERE schemaname = 'public'
ORDER BY pg_total_relation_size('public.'||tablename) DESC
LIMIT 10;

-- Purger les métriques de plus de 90 jours
DELETE FROM device_metrics_history
WHERE time < NOW() - INTERVAL '90 days';

-- Récupérer l'espace disque après purge
VACUUM ANALYZE device_metrics_history;
